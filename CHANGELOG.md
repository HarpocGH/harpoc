## [Unreleased]

### Added

- **Process execution context** for `use_secret` — process-mediated injection alongside HTTP. The vault runs a command as a subprocess (no shell) with the credential in its environment, then captures and sanitizes stdout/stderr. Carries the thesis's output-channel leakage analysis and the I2b invariant.
- Per-secret injection policy (`injection_policies` table, migration 006, KEK-encrypted): a **URL allowlist** for request-mediated injection (optional; re-validated on each redirect hop) and a **command allowlist** for process-mediated injection (fail-safe deny, pinned to a resolved absolute path)
- Output sanitization for process output — redacts the credential and its base64 / base64url / hex / percent-encoded forms
- Clean-environment injection with a controlled PATH; output size cap and execution timeout
- CLI: `harpoc secret use` (`--action http | process | mcp | database | git | ssh`) and `harpoc secret allow` (set/show injection policy)
- REST: `GET` / `PUT /api/v1/secrets/:handle/injection-policy` (trusted administrative path)
- SDK: `setInjectionPolicy` / `getInjectionPolicy`
- Six error codes: `URL_NOT_ALLOWED`, `COMMAND_NOT_ALLOWED`, `PROCESS_SPAWN_FAILED`, `PROCESS_TIMEOUT`, `PROCESS_OUTPUT_LIMIT`, `INVALID_PROCESS_CONFIG`
- **MCP proxy context** for `use_secret` (`action.type: "mcp"`) — the vault as a transparent MCP proxy: forwards a single tool call (`{ server, tool, arguments }`) to a downstream MCP server, authenticating via **stdio** (downstream server spawned with the credential in a clean environment; launch command validated against the command allowlist — fail-safe deny, pinned absolute path) or **Streamable HTTP** (`Authorization: Bearer` injection; URL allowlist + SSRF/DNS validation on every outbound request). No secret value ever enters the agent's context.
- Per-secret downstream MCP server configuration (`mcp_servers` table, migration 007, KEK-encrypted), set via CLI `harpoc secret mcp-server` or REST `GET`/`PUT`/`DELETE /api/v1/secrets/:handle/mcp-server` — never via an MCP tool; `action.server` must match the configured `server_name`
- Downstream connection lifecycle (`McpConnectionRegistry`): spawn on first use, reuse across calls, terminate on every seal path (lock/destroy/session expiry/key wipe); a crash fails visibly (`MCP_SERVER_CRASHED` with exit code and signal), is audit-logged, and never auto-respawns — respawn happens on the next invocation; credential/config staleness is detected via SHA-256 fingerprints and triggers an audited terminate + fresh spawn
- Custom `StdioChildTransport` for spawning downstream servers — the MCP SDK's own stdio transport is not used because cross-spawn shell-wraps commands on Windows and force-inherits the environment; the SDK (`@modelcontextprotocol/sdk`) is now a `core` dependency. Tool results are sanitized across every string leaf (content + structured content) and size-capped
- Three audit events (`mcp.spawn`, `mcp.crash`, `mcp.terminate`) and six error codes: `MCP_SERVER_NOT_CONFIGURED`, `MCP_SERVER_MISMATCH`, `MCP_CONNECT_FAILED`, `MCP_SERVER_CRASHED`, `MCP_PROTOCOL_ERROR`, `MCP_TIMEOUT`
- **Database, Git and SSH contexts** for `use_secret` (`action.type: "database" | "git" | "ssh"`), completing the thesis's six-context taxonomy. Database (request-mediated): in-vault connection assembly for PostgreSQL (`pg`) and MySQL/MariaDB (`mysql2`) behind a lazily-loaded adapter, TLS with server-certificate verification by default and an audited per-secret `tls_mode: "disable"` opt-out, host:port allowlist + SSRF pre-check; the secret value is `username:password`. SSH (process-mediated): an ephemeral **in-process** ssh-agent (`ssh2`; unix socket on POSIX, named pipe on Windows) so the private key never touches disk, spawning the real `ssh` with strict host-key checking against pinned keys. Git (both mechanisms, derived from the repository transport): HTTPS authenticates via a vault-authored `GIT_ASKPASS` helper (credential through the child environment, never argv), SSH reuses the ephemeral agent via `GIT_SSH_COMMAND`; command-executing transports (`ext::`, `file:`, `fd::`, `git+`) and dangerous arguments (`-c`, `--config`, `--upload-pack`, `--receive-pack`, `--exec`) are rejected.
- `host_allowlist` on the injection policy for host targets — fail-safe deny for SSH and Git-over-SSH, optional for database — and per-secret endpoint-authentication pins (database TLS policy, SSH host keys) in a `connection_configs` table (migration 008, KEK-encrypted), set via CLI `harpoc secret connection` or REST `GET`/`PUT`/`DELETE /api/v1/secrets/:handle/connection-config`
- Fourteen error codes for the three contexts: `HOST_NOT_ALLOWED`, `DB_CONNECTION_FAILED`, `DB_QUERY_FAILED`, `DB_TLS_REQUIRED`, `UNSUPPORTED_DB_ENGINE`, `INVALID_DATABASE_CONFIG`, `SSH_CONNECT_FAILED`, `SSH_HOST_KEY_MISMATCH`, `SSH_AGENT_FAILED`, `SSH_NOT_CONFIGURED`, `INVALID_SSH_CONFIG`, `GIT_OPERATION_FAILED`, `GIT_UNSUPPORTED_TRANSPORT`, `INVALID_GIT_CONFIG`
- **Streamable HTTP transport for the vault's own MCP server** — `harpoc server start --mcp-http [--mcp-http-port 3001]` (combinable with `--mcp` and `--rest`) and `harpoc-mcp --http` serve the full MCP tool/resource surface at `/mcp`. Every request requires a vault-issued JWT (`Authorization: Bearer`, same scoped tokens as REST; no tokenless mode over HTTP); the session is pinned to the initializing token via SHA-256 fingerprint and re-verified on every request, so expiry/revocation apply mid-session. DNS-rebinding protection on loopback binds; per-session `McpServer` instances share one process-wide rate limiter

- **Out-of-band value collection for `create_secret` / `rotate_secret`** — the thesis's full channel chain. Channel 1, URL-mode elicitation: when the client declares the `elicitation.url` capability, the vault serves a one-time local form (256-bit single-use token, timing-safe compare, 5-minute expiry, JS-free page) and the browser posts the value directly into the vault process — it never traverses the MCP channel. Channel 2, controlling-terminal prompt (stdio launches only, `enableTtyPrompt`): a masked prompt opened directly on `/dev/tty` (POSIX) or `CONIN$`/`CONOUT$` (Windows) — the server's stdin carries JSON-RPC and is never read; unavailable terminals degrade gracefully. Channel 3, deferred/pending (CLI `harpoc secret set` / `harpoc secret rotate`), unchanged. `rotate_secret` now performs the rotation when a value is collected

- REST API bind address is configurable: `harpoc server start --rest --host <address>` (default `127.0.0.1`, per thesis §4.1; non-loopback binds log a warning)

- **Per-secret HTTP `response_mode`** (thesis §4.5.2) — `full` | `filtered` | `status_only` on the injection policy (default `filtered`), plus an optional per-invocation `response_mode` override on the HTTP action that may only equal or **tighten** the policy mode. A loosening attempt is rejected with the new `RESPONSE_MODE_NOT_ALLOWED` error *before* the request executes, and audited with the requested and policy modes. `status_only` returns the outcome alone — the status code plus headers named in the new `response_header_allowlist` policy field (matched case-insensitively) — and the response body is **never read** (`body.cancel()`), so the echo channel is structurally absent rather than filtered. `filtered` now redacts the credential's base64url, hex and percent-encoded forms in addition to raw + base64, matching the other execution contexts; `full` is the policy-gated raw opt-out (vault-layer redaction skipped; the pattern-based interface guard at the REST/MCP boundaries remains active). CLI: `secret allow --response-mode <mode> --response-header <name>`, `secret use --response-mode <mode>`; REST and SDK carry the new policy fields. No DB migration — the KEK-encrypted policy blob is extended; older blobs default to `filtered`.

- **Interpreter acknowledgement for command allowlists** (thesis §4.5.3) — adding a known interpreter (shells, script hosts, `-c`/`-e` runtimes, package runners like `npx`/`uvx`; curated basename list in `@harpoc/shared`, matched case-insensitively with `.exe` and version suffixes stripped, so `Python3.12.EXE` gates as `python`) to a secret's command allowlist requires an explicit per-operation acknowledgement: CLI `secret allow --acknowledge-interpreter`, REST `acknowledge_interpreters`. An unacknowledged addition is refused with the new `INTERPRETER_NOT_ACKNOWLEDGED` error *before* the policy is written; refusal and acknowledged grant are both audited (`policy.interpreter_refused` / `policy.interpreter_acknowledged`, naming the entries). Only newly added entries gate — re-asserting a stored interpreter never re-prompts — and the gate composes transitively: a stdio MCP downstream server launched via `node`/`npx` needs the acknowledged allowlist entry

- **Secret-name patterns in token scopes** — the token's `secrets` claim (CLI `auth token --secrets`) now holds full-anchored name patterns with `*` wildcards (e.g. `db-*`; case-sensitive, `*` the only meta-character) instead of exact names. Patterns are validated at token creation (`INVALID_SECRET_NAME` before signing) and enforced by one shared linear matcher at all four sites — MCP scope checks and list filtering, REST scope middleware and list filtering; the project dimension stays exact-match

- **Windows session-key protection (DPAPI)** — the session file's `session_key` is wrapped with DPAPI at rest on Windows (scheme-tagged via a `key_protection` field; absent means none, so existing session files stay readable). Reads fail closed — a tampered, foreign-user, or scheme-mismatched file requires a fresh `unlock`; a write-time wrap failure falls back to file permissions with a CLI warning; `HARPOC_SESSION_KEYSTORE=off` opts out for new writes. The DPAPI call goes through the OS-shipped PowerShell host (pinned path, `shell:false`, base64 over stdin/stdout, no cmdlets or `Add-Type`) — no native addon; sliding-TTL updates operate on the stored form, so the keystore roundtrip (~150 ms) is paid once per session load. macOS Keychain and Linux Secret Service remain future work

- **Tamper-evident audit log** — audit `detail` blobs are now AAD-bound to their own row (v2, no legacy fallback: blobs written before the change render as unreadable detail instead of being silently trusted), and every audit row carries an HMAC chained over its predecessor (migration 010). The new `harpoc audit verify` command walks the chain and exits non-zero on a break. Residual: truncation of the chain tail is only detectable against an external anchor

- **CLI OAuth flows — `harpoc oauth connect` / `status` / `refresh` and `server start --oauth-refresh`** — the OAuth 2.1 proxy is now wired to shipped entry points. `harpoc oauth connect <name>` creates the OAuth secret and runs the provider flow end-to-end over the unlocked vault (PENDING → ACTIVE): authorization code + PKCE by default — the authorization URL is **printed** (to stderr) and `--open` additionally launches the browser — `--device` for the headless device-code flow (provider presets supply the device endpoint), `--client-credentials` for machine-to-machine. The client secret never travels via argv: the `HARPOC_OAUTH_CLIENT_SECRET` environment variable wins, otherwise a hidden prompt (empty = public client; refused for `--client-credentials`, which requires one). Ctrl+C cancels pending flows and leaves the secret PENDING for a re-run. `harpoc oauth status <handle>` reports token health without sensitive fields; `harpoc oauth refresh <handle>` forces a refresh immediately. `harpoc server start --oauth-refresh` runs the background token-refresh scheduler in the long-lived server process — valid standalone as a refresh daemon — surfacing per-token refresh failures as CLI warnings; shutdown stops the scheduler before the vault seals

- **File-based secret import and encrypted-key decryption at import** — `harpoc secret set` and `harpoc secret rotate` accept `--from-file <path>` (byte-exact, 1 MiB cap), enabling multi-line values such as PEM private keys, which the single-line hidden prompt cannot carry. When the value is a passphrase-protected **PKCS#8 or legacy-PEM** private key, the vault prompts once for the passphrase, decrypts the key **in memory** (`node:crypto` only — no new dependencies, no temp files), and stores the plaintext key under the vault's own encryption; the passphrase is never persisted, and a wrong passphrase exits non-zero with the new `KEY_PASSPHRASE_INVALID` without writing anything. `--no-decrypt` stores encrypted material verbatim. Encrypted **OpenSSH-format** keys remain unsupported (bcrypt-pbkdf is outside `node:crypto`) and are refused before any prompt with the new `ENCRYPTED_KEY_UNSUPPORTED`, whose message carries the per-key-type recovery: RSA/ECDSA convert in place with `ssh-keygen -p -f <keyfile> -m PKCS8` (passphrase kept; the converted key then imports with the prompt), while for ed25519 — where ssh-keygen silently ignores `-m PKCS8` — the passphrase is stripped with `ssh-keygen -p -f <keyfile> -N ''` and the resulting unencrypted OpenSSH container imports as-is. Use-time errors for encrypted keys now point at the same recovery instead of a dead end. The `use_secret` path is unchanged

- **Audit-chain external anchor — `harpoc audit anchor` and `verify --anchor`** — closes the tamper-evident audit log's documented residual: the HMAC chain catches row tampering, detail swaps and interior deletion, but deleting the newest N rows leaves a valid shorter chain. `harpoc audit anchor` exports the chain's tail link as a small JSON document (`format: "harpoc-audit-anchor/1"`; vault id, row id, timestamp, link HMAC in hex — nothing sensitive, the HMAC is already stored in plaintext) to stdout for piping off-host, or to a file with `--out`. `harpoc audit verify --anchor <file>` then additionally asserts the anchored row still exists with exactly that link — tail truncation and database rollback fail the check (exit 1), while rows appended after the anchor keep verifying; an anchor from a different vault is rejected by vault id. `verify` now always prints the current tail link for eyeball comparison and re-anchoring. **The anchor only means something stored off-host** (another machine, a sync target the attacker cannot write, or paper): an attacker with database write access can likely also modify sibling files, so the vault deliberately has no anchor-storage feature — and rows appended after the newest anchor are unprotected until the next one is taken, so re-anchor regularly. Engine surface: `getAuditChainTail()` and `verifyAuditChain({ anchor })`. No migration, no new error codes, no new audit events

### Changed

- The ephemeral in-process ssh-agent (SSH and Git-over-SSH contexts) is **reimplemented on `node:crypto`**, and `ssh2` — with its transitive `bcrypt-pbkdf` and `tweetnacl` — is **removed from `core`**. The vault now implements the OpenSSH agent protocol, SSH private-key parsing (traditional PEM and the `openssh-key-v1` container, via a hand-walked container → JWK → `node:crypto` KeyObject) and signature generation itself, so no third-party cryptographic library loads into the vault process — `node:crypto` plus Argon2id are the only cryptographic components (restores the single-exception property in the security model). Supported key types: **unencrypted** Ed25519, RSA and ECDSA (nistp256/384/521); RSA signs with `rsa-sha2-256`/`-512` per the agent's flags (never legacy SHA-1). Encrypted keys are rejected with a clear error at use time (the vault provides encryption at rest, and supporting them would re-introduce `bcrypt-pbkdf`; passphrase-protected PKCS#8/legacy-PEM keys are decrypted at import — see the encrypted-key import entry under Added); the prior `ssh2` path already required unencrypted keys, so realistic usage is unchanged. Validated against real OpenSSH (`ssh-add` parses the agent's identity blob) and by `node:crypto` signature verification across all key types and both encodings. The agent's public surface (`EphemeralSshAgent.start/authSock/dispose`) is unchanged; the SSH/Git injectors are untouched.
- The `use_secret` action dispatch and the interface-boundary result sanitizer are now **compile-time exhaustive**: the engine's six-context branch became a `switch` whose `default` arm funnels into a `never`-typed `assertNever`, and `sanitizeUseSecretResult` gained the same guard. Adding a seventh action or result type without an explicit arm is now a type error instead of a silent compile (verified by temporarily extending both unions). Runtime behavior for type-evading callers is unchanged for actions (`INVALID_INPUT`, same message) and **hardened** for results: an unknown result shape previously fell through the sanitizer silently — returning it unsanitized — and now throws `INVALID_INPUT` instead
- The shared type system is now **schema-derived**: every externally-validated shape — the six `use_secret` actions and their union, `InjectionConfig`, the injection policy, the downstream MCP server config, the connection configs, the session file and the OAuth provider config — has its TypeScript type derived from its Zod schema (`z.infer` beside each schema in `schemas.ts`), replacing the hand-written twin interfaces in `types.ts`; the validator and the type can no longer drift apart. Consequences: `InjectionConfig` is now the discriminated union the schema always enforced (`header` requires `header_name`, `query` requires `query_param`) instead of a loose all-optional interface, and `InjectionPolicy` now describes the policy the vault actually loads and returns (every field present, defaults applied) while the new `InjectionPolicyInput` (every field optional) is what `setInjectionPolicy` accepts across engine and SDK — matching the implementation's long-standing defaulting behavior. TypeScript-facing only; wire formats and stored blobs are unchanged
- **Breaking (SDK):** the SDK's input types are schema-derived too. `CreateSecretInput`, `GrantPolicyInput` and `HealthResponse` in `@harpoc/sdk` now derive from the shared Zod schemas (`createSecretInputSchema` → `CreateSecretRequest`, `accessPolicyInputSchema` → `AccessPolicyInput`, new `healthResponseSchema`) instead of duplicating them as hand-written interfaces. The inputs follow the wire shape: `GrantPolicyInput` is now `{ principal_type, principal_id, permissions, expires_at? }` (previously camelCase) and `CreateSecretInput` takes `expires_at` (previously `expiresAt`) — consistent with every other type the SDK exposes; `value` stays `Uint8Array` (the REST client base64-encodes it in transit) and the SDK remains zod-free (type-only imports). The shared `createSecretInputSchema` now covers the full create body (`value` as validated base64, `expires_at`), so `POST /api/v1/secrets` rejects malformed base64 in `value` with 400 instead of silently best-effort decoding it
- The MCP `create_secret` tool validates its `injection` argument against the canonical `injectionConfigSchema` instead of a hand-rolled looser copy: a malformed config such as `{type: "header"}` without a `header_name` is now rejected at the tool boundary (as REST always did) instead of being stored and failing only at first use
- Windows batch files (`.cmd`/`.bat`) are now excluded by the **vault's own command resolution** instead of by the Node runtime. The resolver no longer probes `.cmd`/`.bat` when resolving bare command names, and any command whose symlink-resolved path is a batch file — including an explicitly allowlisted absolute path — fails with `COMMAND_NOT_ALLOWED` before anything is spawned. Rationale: spawning a batch file with `shell:false` is cmd.exe interpretation in disguise — Node versions before the CVE-2024-27980 fix silently wrapped such spawns in cmd.exe, an argument-injection surface that collapses the no-shell property and would let a `deploy.cmd` allowlist entry grant shell capability without ever tripping the interpreter-acknowledgement gate; patched versions refuse with EINVAL at spawn time, i.e. courtesy of the runtime. The vault now owns the exclusion deterministically on every Node version, at the same choke point that covers process actions, MCP stdio downstream launches and ssh/git resolution. Batch scripts remain runnable the explicit way: allowlist `cmd` with the interpreter acknowledgement and run `cmd /c script.cmd`. The exclusion is pinned by Windows-gated tests, including the symlink smuggle vector (a link named like a safe executable resolving to a `.cmd`); the symlink case requires file-symlink privilege (Developer Mode, elevation, CI runners) and skips where absent
- The session TTL now slides on **authenticated use**, not on process liveness. Every authenticated engine operation passes through a throttled expiry slide (at most one session-file write per 30 s per process — the cadence the background monitor used to produce), and `use_secret` slides again on completion, so a long-running action ends with a fresh idle window. The 30 s monitor no longer extends the session — it only enforces expiry: an idle-but-running MCP/REST server now seals about 15 minutes after its last use instead of keeping the session alive to the 24 h ceiling by mere liveness, and short-lived CLI commands — which exited long before the first monitor tick and therefore never slid at all — now genuinely extend the window on every command. Slides operate on the stored form (no keystore roundtrip), never fail the operation they ride on, and `lock()` settles an in-flight slide before the secure erase so a racing rename cannot resurrect the session file
- `harpoc secret allow` now **merges** into the stored policy: flag groups you omit keep their current values (previously every call replaced the whole policy, so a `--url`-only update wiped the command/host/env lists — and would have silently reset a `status_only` response mode back to `filtered`). The new `--clear` flag resets the policy to defaults before applying the other flags. REST `PUT /api/v1/secrets/:handle/injection-policy` remains a whole-policy replace; policy writes are audited with the effective response mode.
- The HTTP redirect loop now cancels the unread body of each followed redirect hop instead of leaving the connection pinned until garbage collection
- `wipeBuffer` now zeroes buffers (`Buffer.fill(0)`) instead of random-filling, matching the thesis's §4.6 memory-hygiene discipline; the session-file secure erase keeps its intentional random-byte overwrite
- **Breaking:** `use_secret` request shape changed from `{ request, injection, follow_redirects }` to `{ action }`, a discriminated union on `action.type` (`http` | `process` | `mcp` | `database` | `git` | `ssh`), across MCP, REST and SDK. Responses are discriminated too (`HttpResult` | `ProcessResult` | `McpResult` | `DatabaseResult` | `GitResult` | `SshResult`). The existing HTTP path is now `action.type: "http"`.
- The vault format-version compatibility check compares version components **numerically** and fails closed: a lexicographic comparison orders `1.10.0` before `1.2.0`, silently accepting a newer vault across a digit boundary; a malformed version string (empty, non-numeric component) now refuses to open instead of guessing
- The sliding session TTL is enforced for engines created by `init`/`unlock` too, not only for sessions re-attached via `loadSession`; a new `sessionTtlMs` engine option (capped at the 24 h ceiling) exposes the window as an operator knob, and an expiry-racing TTL slide can no longer resurrect an already-expired session file
- The shared OAuth endpoint schemas accept loopback HTTP (`localhost` / `127.0.0.1` / `[::1]`) in addition to HTTPS, matching the runtime SSRF validator that guards every OAuth fetch; non-loopback HTTP remains rejected at the schema boundary

### Removed

- The legacy unlock fallback that HKDF-derived the JWT/audit keys from the master key when a vault carried no wrapped keys, along with the one-time migration that back-filled them. The key hierarchy now has exactly one instantiation — random JWT/audit keys wrapped with the KEK — and a vault missing `wrapped_jwt_key`/`wrapped_audit_key` fails closed with `VAULT_CORRUPTED` before any key derivation runs, pinned by tests (missing keys refuse to unlock; a tampered wrapped key is rejected, never bypassed by a fallback). The fallback dated to the 2026-03-04 security-review migration from derived to random keys; every vault created since — including everything v1.0.0 produces — already stores wrapped keys, so no supported vault is affected. The now-dead `HKDF_INFO_JWT_SIGNING`/`HKDF_INFO_AUDIT` constants are gone (`HKDF_INFO_SYNC` stays, reserved for cloud sync)

### Fixed

- **Security (code review 2026-07-07) — hidden prompt echoed the master password.** `promptHidden` (CLI `init`, `unlock`, `secret set`, `secret rotate`) attached a `readline` interface to a TTY output while a manual raw-mode scanner did the actual read; in a real terminal readline's line-editing echoed every keystroke, printing the password in cleartext (and leaving it in scrollback). The vestigial `readline` interface is removed — the raw-mode scanner is the sole reader, so nothing echoes. (Non-TTY/piped input was never affected, which is why prior tests missed it; a TTY-flagged output-stub test now guards it.)
- **Security — re-running `harpoc init` on an existing vault destroyed all secrets.** `initVault` unconditionally rewrapped the KEK, orphaning every secret's DEK (wrapped under the old KEK) with no recovery and no warning. `init` now refuses when a vault already exists — `initVault` throws the new `VAULT_ALREADY_EXISTS` error (HTTP 409) and the CLI checks for the vault before prompting for a password, pointing at `harpoc unlock`. There is intentionally no `--force`: re-initialization is irreversible, so starting over is the explicit act of deleting the vault directory.
- **Security — `GET /api/v1/health/expiring` and MCP `check_secret_health` leaked out-of-scope secret metadata.** Both aggregated over the full secret list with no token scoping, so a project- or name-pattern-scoped token (or, for the REST route, a token without `list` permission) could enumerate every secret's name/project/expiry across the vault; the REST route's `days` was also unbounded. The REST route now enforces the `list` permission, applies the token's project + name-pattern filtering (parity with `GET /secrets`), and bounds `days` to `1..365`; the MCP tool aggregates over `scopeGuard.filterByScope(...)`.
- **Security — ReDoS in the secret-name scope matcher.** `matchesSecretNamePattern` compiled a token's `secrets` pattern to a regex by joining wildcard segments with `.*`, which backtracks exponentially: a token minted with `secrets: ["*a*a*a…"]` made every subsequent scope check burn unbounded CPU (~15 s at 10 wildcards, effectively unbounded at the 255-char limit) — a single-token denial of service of the vault process. Matching is now a linear prefix/suffix + ordered-`indexOf` segment walk with no backtracking; behavior is verified identical to the previous matcher across an exhaustive differential test.
- **Robustness — an invalid secret name broke `listSecrets` vault-wide.** `createSecret` inserted the row before validating the name, so `harpoc secret set "bad.name"` left an unaddressable phantom row that made every subsequent `list` (CLI, MCP, REST) throw. Name and project are now validated before anything is written (`createOAuthSecret` inherits the fix).
- **Security — database SSRF was DNS-rebinding-bypassable.** The database injector resolved and SSRF-checked the host but then passed the raw hostname to the driver, which re-resolved it — so a name that passed the pre-flight lookup as a public address could resolve to an internal one at connect time (and with `tls_mode: "disable"` no certificate check would catch it). The validated IP is now the dial target: `pg` connects to the pinned address with `ssl.servername` set to the hostname (SNI and certificate verification stay on the name), and `mysql2` dials the pinned IP through a custom socket factory with `verifyIdentity: true` (its hostname check is otherwise a no-op). Unpinned targets (literal IPs, loopback) are unchanged.
- **Security — the MCP-over-HTTP proxy had no connection pinning and followed downstream redirects unchecked.** `validatingAuthFetch` validated the URL but then used the global `fetch` with default redirect-following, leaving a DNS-rebinding window and letting a compromised downstream server redirect the credential-bearing request to an arbitrary address. Every outbound request now routes through a per-connection pinned `undici` `Agent` (reusing the HTTP injector's validated-address lookup), and downstream redirects are refused outright — `redirect: "manual"`, any 3xx cancels the body unread and throws `REDIRECT_POLICY_VIOLATION` (the downstream endpoint is admin-configured and exact, and redirect semantics break JSON-RPC anyway).
- **Security — MCP `create_secret` skipped the secret-name scope check.** The tool called `scopeGuard.checkAccess(permission, project)` without the name, so a token scoped to `secrets: ["api-*"]` could create `admin-backdoor`, and a project-scoped token could create a global (project-less) secret — both outside the declared scope. The name is now passed to `checkAccess`, matching the REST create route; a project-scoped token must supply its project on create.
- **The create-time `injection` config was accepted and silently discarded — the field is removed.** `harpoc secret set --injection/--header-name/--query-param` and the MCP `create_secret` tool's `injection` parameter promised a stored default injection configuration, but nothing ever persisted or read it: users believed they had configured injection behavior that did not exist, while every `use_secret` call continued to require its own per-call `injection` object (unchanged, and still required). The dead field is gone from the CLI flags (now rejected as an unknown option instead of ignored), the MCP tool schema, the REST create body (a legacy body carrying it is still accepted — the key is stripped, matching the previous effective behavior) and the engine/SDK input types. Setting injection-related configuration at create time via MCP also contradicted the rule that such configuration is set only through the trusted admin path (CLI/REST `secret allow`); a genuine per-secret default-injection feature, if ever wanted, belongs on the injection policy
- **Security — denied secret access and lazy-expiry transitions left no audit trail.** Every engine operation logged its audit entry only after success, so a scoped token probing revoked, expired or nonexistent secrets — the reconnaissance an audit trail exists to catch — was completely invisible, and the lazy-expiry status write (ACTIVE → EXPIRED on access) was an unaudited vault mutation with a `secret.expire` event type that had been defined since v1.0 but never emitted. Denied reads, rotations, revocations and `use_secret` value resolutions now produce a `success: false` entry carrying the error code (the same pattern as the existing URL-allowlist denial), and the lazy-expiry transition emits `secret.expire` exactly once — at both transition sites (regular secrets and OAuth access-token reads)
- **Security — the POSIX session file was world-readable during the write.** The session file — which on POSIX holds the raw session key that unwraps the KEK (keystore wrapping is Windows-only) — was created with default permissions (typically 0644) and tightened to owner-only 0600 only *after* fsync and rename, with a chmod failure silently swallowed: another local user could read the key during the write window, or indefinitely after a failed chmod. The temp file is now created with mode 0600 from the first instant (the rename preserves it, and umask can only tighten it), so the window no longer exists; the trailing chmod remains as a repair whose failure now fires the session-protection fallback callback, so the CLI warns instead of staying silent
- **Security — the MCP-over-HTTP body-size cap only protected the initialize request.** The vault's own 4 MiB reader guarded initialize, but requests carrying a valid `Mcp-Session-Id` were handed unread to the SDK transport, which buffers the body with no size limit — any client holding a valid session (any valid token) could POST an arbitrarily large JSON body and exhaust vault memory. Every body-carrying request now goes through the capped reader, and the parsed body is handed to the SDK so its unbounded read never runs. The reader also no longer destroys the request stream on an oversized body (which could reset the connection before the `413` was delivered); it drains and discards the remainder so the error response reaches the client
- **`harpoc audit verify --json` silently ignored the flag.** The parent `audit` command declares `--json` too, and commander binds a post-subcommand `--json` to the parent — so `verify --json` always took the plain-text branch. The subcommands now read the flag through the merged option view (`optsWithGlobals`), and `audit verify --json` / `audit anchor --json` behave as documented
- **Security — OAuth token refresh failures were invisible in the audit trail.** Every failure branch of the refresh path threw before the audit line, so a scheduler repeatedly retrying a revoked grant left no trace. Failed refreshes now produce an `oauth.refresh` entry with `success: false` and the error code, matching the denied-access audit pattern
- **Security — the Git dangerous-argument denylist missed `--template` (clone-time code execution).** `action.args` is agent-controlled and spliced into the git argv; the denylist rejected `-c`/`--config`/`--upload-pack`/`--receive-pack`/`--exec` but not `git clone --template=<dir>`, which copies the template directory's `hooks/` into the new repo and runs the non-`.sample` hooks during the clone — local code execution for an agent that already holds a file-write primitive. The denylist now also rejects `--template` and `--separate-git-dir` (matched by name prefix, so both the `--opt=value` and `--opt value` forms are caught), and blocks the `-u` upload-pack shorthand for `clone` only, leaving `push -u` (`--set-upstream`) working.
- CLI hidden prompts (`init`, `unlock`, `secret set`, `secret rotate`) now work with piped/non-interactive stdin. The prompt previously treated every stdin `data` chunk as a single keystroke, so a piped line like `printf 'pw\n' | harpoc unlock` never matched the Enter check — the prompt hung and, on EOF, the process **exited 0 without executing the command**. Input is now scanned per character regardless of chunking, input after the line terminator is pushed back for the next prompt (`printf 'pw\npw\n' | harpoc init` answers both prompts), EOF terminates the line (an empty value still fails loudly), and a prompt can no longer leave the process exiting successfully with nothing done
- Redirect hops are now re-validated against the secret's **URL allowlist**, not just the scheme/SSRF floor. The allowlist previously ran once at the engine against the initial URL only, so with `follow_redirects: "any"` a credential-bearing request could follow a cross-origin redirect to a non-allowlisted origin — and in `same-origin` mode a (credential-stripped) request could still reach one. Every hop is now checked against the same patterns before it executes: a non-matching hop fails with `URL_NOT_ALLOWED` (audited, `success: false`) and the redirect target is never fetched. Empty allowlists remain unenforced (the optional-atop-floor posture is unchanged); with an allowlist configured, hop enforcement applies in **every** `follow_redirects` mode, including path-restricted patterns on same-origin hops
- HTTP DNS-rebinding IP pinning now actually executes. The pin branch was dead code: the validator only ever returned a resolved address for public hostnames — which must be HTTPS — while the branch additionally required plaintext `http:` (loopback-only), so the two conditions were mutually exclusive and **no request was ever pinned**; rebinding protection silently rested on TLS certificate validation alone. The pre-flight lookup now resolves **all** A/AAAA addresses (each checked against the SSRF policy — a single private address among the results blocks the request), and every request — the initial one and each redirect hop after its own re-validation — connects through a per-request `undici` `Agent` whose connection-time DNS lookup serves only the validated addresses. Pinning at the socket layer keeps the URL's hostname intact, so the `Host` header and TLS SNI/certificate validation are unaffected, and the mechanism now covers HTTPS (where it previously could never apply). `ValidatedUrl.resolvedAddress` (singular) became `resolvedAddresses`; `undici` — the engine behind Node's own `fetch` — is now an explicit `core` dependency
- Concurrent session-file writers can no longer collide on a shared temp file: the atomic write-then-rename now uses a per-write temp name (pid + counter). Previously all same-process writers shared `.session.json.tmp.<pid>`, so an expiry slide racing `changePassword`'s session rewrite could fail with `EPERM` on Windows; the engine additionally settles an in-flight slide before rewriting the session
- **`harpoc secret delete` no longer exits 0 without acting on closed/piped stdin.** The confirmation prompt used a readline callback that never fires on EOF, so `printf '' | harpoc secret delete x` exited successfully having neither deleted nor aborted, and `printf 'y'` (no newline) did the same. The prompt now uses the same chunk-independent scanning as the hidden prompts: EOF acts as Enter (`printf 'y'` confirms; an empty answer is the default No), a stream error settles the promise, and every outcome either deletes or prints "Aborted."
- **`harpoc secret connection` no longer silently drops stored endpoint-auth pins.** The command previously rebuilt the whole config from the provided flags, so `--db-tls require` after an earlier `--db-ca-file` discarded the CA pin (silently downgrading database certificate verification to system trust) and any DB flag wiped the SSH `known_hosts` pins. It now merges per field — omitted flags keep their stored values, provided `--known-host`/`--known-hosts-file` flags replace the stored SSH list — with a new `--clear` flag to reset, matching `secret allow`'s semantics
- `harpoc audit --since <unparseable date>` now fails with a clear error instead of silently returning the full unfiltered list (`NaN` was dropped by the store's falsy filter check); `harpoc policy grant --principal-type <invalid>` now fails with a clean validation message instead of a raw SQLite constraint error
- **Bearer tokens no longer have to travel via argv:** `harpoc server start --mcp` and `harpoc auth revoke` read the new `HARPOC_TOKEN` environment variable when `--token` is not given (an explicit flag wins; an ambient variable without `--mcp` is ignored). Command-line arguments are visible to every local process and land in shell history — the env var closes that exposure
- **Security — the username half of `user:password` secrets was never redacted.** Database and Git-over-HTTPS output sanitization redacted only the password: the username — credential material — survived in stdout/stderr, query result rows and error detail. Both halves are now redacted across all output channels (raw + base64/base64url/hex/percent forms); usernames shorter than 3 characters stay unredacted (they would shred unrelated output)
- **Security — SSH host/user argv hardening.** `action.host` was passed as a bare positional to `ssh` with no leading-dash guard and a schema that permitted `-host`-shaped values (option smuggling surface; no working RCE — the charset forbids `=`, space and `/`). Three layers now close it: the schema anchors the first character to an alphanumeric, the injector refuses `-`-prefixed host/user before any other work, and the argv carries a `--` end-of-options separator before the host positional
- Host:port validation now bounds the embedded port: `db.example.com:70000` is rejected at the schema (`hostPattern`) and again by the database injector's `parseHostPort` (defense in depth), matching the standalone `port` field's 1–65535 range
- HTTP action and MCP downstream-server URLs now reject non-http(s) schemes (`javascript:`, `file:`, `ftp:`, …) at the schema boundary instead of one layer down in core's SSRF validator; caller-supplied HTTP headers are validated at the boundary (name charset, 8 KiB value cap with CR/LF/NUL rejected, 64-header cap)
- The SDK `RestClient` now times out (default 30 s, configurable via `timeoutMs`) instead of hanging forever on an unresponsive vault server, and `DirectClient.revokePolicy` enforces the same policy-belongs-to-secret ownership check as the REST route (contract parity; previously the handle argument was ignored)
- **OAuth `CallbackServer` no longer leaks its timer and listener:** every terminal branch (success, provider error, missing params, invalid state) now clears the up-to-5-minute timeout, settles the callback promise exactly once, and closes the server after the browser response is flushed; the timer is `unref`'d so a pending flow cannot block process exit
- **Device-code background polling is now abortable and observable:** `OAuthManager` tracks each background poll's `AbortController` (`cancelFlow(secretId)` / `cancelPendingFlows()`), the poll-loop timer is `unref`'d (a pending device flow no longer blocks CLI exit for up to 15 minutes), and a background completion failure — previously swallowed — is surfaced through the new `onBackgroundFlowError` callback option
- **Failed OAuth refreshes now back off exponentially** instead of being retried every scheduler tick forever: a per-secret in-memory quarantine doubles the retry interval per consecutive failure (capped at 1 hour), clears on success, and is bypassed by an explicit `refreshNow`
- **OAuth provider presets: `client_secret_basic` support and the Google device endpoint.** The provider config accepts `token_endpoint_auth_method` (`client_secret_post`, the default, or `client_secret_basic` — `Authorization: Basic` with form-urlencoded credentials, kept out of the request body) honored at the authorization-code exchange, client-credentials grant and device-code poll; the `google` preset gains `device_authorization_endpoint`, so `google` + `device_code` works out of the box. (Residual: the core refresh path still authenticates via `client_secret_post` — the stored token record carries no auth-method column.)
- The MCP server's per-secret rate-limit bucket eviction actually evicts now: the sweep tested the stored token count against the limit, but every access decrements after refill and idle buckets are never refilled, so no bucket ever qualified and the map grew unboundedly across distinct secrets. The sweep now computes the idle-time refill, evicting buckets whose idleness would fully refill them
- **Security — concurrent `createSecret` calls could both pass the uniqueness check (TOCTOU).** The name-uniqueness check and the insert now run in one transaction backed by a partial unique index over live secrets' `name_hmac` (migration 009), so the race loses cleanly with `DUPLICATE_SECRET` instead of storing two secrets under one name
- Key-buffer hygiene: decryption and unwrap failure paths now wipe intermediate key material in `catch`/`finally`, a repeated `unlock` wipes the previous key set first, and `unlock()` no longer leaks an open store handle on its failure paths; `changePassword` now shares `unlock`'s lockout accounting and `INVALID_PASSWORD` semantics (previously divergent)
- **Security — OAuth token and device endpoints were fetched without URL validation.** Every oauth-proxy fetch — the authorization-code exchange, the client-credentials grant, and the device-flow start and poll — now runs the vault's URL validator (HTTPS requirement + SSRF policy) before the request, placed ahead of the flow's own error handling so `SSRF_BLOCKED`/`URL_HTTPS_REQUIRED` surface unrewrapped; loopback HTTP remains allowed for local providers
- **Concurrent OAuth refreshes no longer replay the same refresh token.** A scheduler tick racing an on-demand refresh sent the same `refresh_token` twice — providers with rotation detection revoke the whole token family on replay. Refreshes are now coalesced per secret (all callers share a single in-flight refresh, cleared on completion and on key wipe), and the scheduler skips a secret whose refresh is already running
